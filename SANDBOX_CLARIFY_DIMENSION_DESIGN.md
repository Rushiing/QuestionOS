# 沙盘步骤①追问维度库设计方案

## 背景

沙盘推演的关键在于**步骤①追问的质量**——追问越针对性、越围绕审议室特有的信息缺口，用户就能更快、更好地定义问题，进入步骤②的沙盘推演。

### 前状态问题
- 追问生成完全依赖 LLM 自由生成，缺乏结构化指导
- 同一套 prompt 对所有审议室生效，没有场景感知
- 容易产生重复追问或通用套话（"最想保住什么、最怕失去什么"）
- 无法控制追问的维度和递进逻辑

## 解决方案：审议室特有的维度库

### 核心设计

为每个审议室（六室）定义 **3~4 个关键追问维度**：

#### 1. BUSINESS（集市 - 商业与战略）
- **核心商业选择**：定价/融资/扩张/转向/入场/退出中的哪一个
- **硬约束清单**：资金量级、融资时限、团队规模、市场窗口期
- **关键信息缺口**：竞争反应、用户需求、成本结构、最怕错的假设

#### 2. ENGINEERING（锻造坊 - 工程与架构）
- **技术选型权衡**：性能/可维护性/开发速度/扩展性
- **成本维度最紧**：开发时间 vs 维护成本 vs 计算资源 vs 学习成本
- **最小验证路径**：如何快速验证架构决策的可行性

#### 3. LIFE_CROSSROADS（神谕所 - 人生十字路口）
- **具体抉择**：转行/留任/创业/深造/地理迁移中的哪一个
- **硬边界与底线**：家庭、财务、身体、时间的绝对约束
- **价值驱动**：安全感/自由/成长/影响力中最在乎什么
- **选择可逆性**：如何衡量后悔的成本和多久后才无法回头

#### 4. RELATIONSHIP（火炉边 - 关系与家庭）
- **反复冲突那一点**：每次吵到哪句话就卡住、是表面还是深层
- **双方真实诉求**：对方真正期待什么、你真正想要什么
- **过去化解记忆**：之前成功化解过吗、是怎么做的

#### 5. PSYCHOLOGY（诊疗室 - 心理韧性与行为）
- **触发节点**：什么场景/时间/人/事件最容易激发问题行为
- **恶性循环**：拖延→焦虑→睡眠差→更难开始，哪一环最先打破
- **最小可改一项**：睡眠/运动/社交/工作时段中当下最可能改的是什么

#### 6. CREATIVE（工作坊 - 创作与表达）
- **创作卡点**：立意不清/结构混乱/完成度不够/害怕发表中的哪一个
- **读者契约**：希望读者得到什么体验/思考/感受
- **最小可展示版本**：70% 完成度的版本能否分享

#### 7. GENERAL（广场统筹 - 跨域或信息不足）
- **决策对象钉死**：「迷茫」太宽，你到底在决策什么
- **约束清单**：哪条约束最硬、哪条有灵活空间
- **最坏情景下最先断**：失败了最怕失去什么

## 实现细节

### 1. 新增类
**`SandboxClarifyDimensionLibrary.java`**
- 为每个审议室提供维度列表
- `generateDimensionHint(SandboxDeliberationScene scene)` 生成 prompt 段落提示

### 2. 修改 `MainCalibrateAgent`
- **增强 `SANDBOX_STEP1_CLARIFY_PROMPT`**：加入维度库的参考提示，但不强制枚举
- **修改 `generateSandboxStep1ClarifyFollowup` 签名**：加入 `deliberationScene` 参数
- **修改 `buildSandboxStep1UserPayload` 重载**：在 payload 中注入场景和维度提示

### 3. 修改 `SessionService`
- 调用追问生成时，传入 `classificationSnapshot.scene()` 或显式的 `SandboxDeliberationScene`
- 两处调用均已更新：
  - 问题不够具体时：`generateSandboxStep1ClarifyFollowup(history, null, null)`
  - 已分诊但信心 LOW 时：`generateSandboxStep1ClarifyFollowup(history, classificationSnapshot, sc)`

## 使用效果

### 对用户的影响
1. **追问更针对性**：LLM 不再生成通用模板，而是根据审议室维度生成自然的、具体的追问
2. **信息密度更快达到**：目标是 1~2 轮追问后能进入步骤②，同时保证问题质量足够
3. **减少追问重复**：维度库避免了「最想保住什么」类三件套的重复出现

### 示例流程
```
用户："我在考虑创业"
↓
[步骤①] 问题过泛，先追问
LLM 看到 scene=BUSINESS 的维度提示
追问："在融资/产品-市场契合/团队组建中，现在最卡的是哪一个？"
↓
用户："产品还没有市场反馈"
↓
[步骤①] 问题更具体了，但仍需一轮澄清
LLM 看到「硬约束清单」维度
追问："从融资或运营成本的角度，这次验证最多能消耗多少时间/资金？"
↓
用户：[回答约束信息]
↓
[语义点火] 判定问题已足以启动沙盘 → [步骤②] 进入六间审议室
```

## 后续优化方向

1. **多轮追问的递进树**（Phase B 方案）：当前是按维度选择，可进一步优化为 scenario_confirm → 具体维度 → synthesis 的递进链
2. **语义点火的场景感知**（可选）：`isSandboxSemanticIgnitionReady` 可加入场景特定的门槛标准
3. **维度库的参与度统计**：记录每个维度在追问中的使用频率，用于持续优化
4. **A/B 测试**：对比有维度库指导 vs 无指导的追问质量和进度
