# QuestionOS vs Agora：沙盘推演的差距分析

## 总体对比

### Agora（理想状态）
- **8 步完整流程**：Evidence → Restate → Round1 → AdaptiveGate → Round2 → Synthesis → Verdict
- **31 个具体思想家**，跨 6 个主题领域
- **黑格尔辩证法**：不仅对立，还有高阶综合
- **高度用户交互**：2 个明确检查点 + 问题溶解检查 + 隐性知识提取
- **多模式**：Full / Quick / Duo / Triad / Custom
- **证据驱动**：每个房间有具体的证据收集策略
- **自适应深度**：根据共识度决定深度

### QuestionOS 当前实现
- **简化的 2 步流程**：步骤① (分诊+追问) → 步骤② (4 角色轮转)
- **4 个固定角色**：Auditor / Value Judge / Risk Officer / Integrator
- **简单轮转**：每轮一个 agent，无立场碰撞
- **有限的用户交互**：主要是步骤①的追问环节
- **单一模式**：都是完整模式，无快速/简化选项
- **会话模式驱动**：分诊结果决定审议室，而非分诊+面板选择

## 逐层差距分析

### 🔴 差距 1：问题理解的完整性

#### Agora 做法
```
Step 2: Problem Dissolution Check
- 识别伪问题（Pseudoproblem）
  例：「我应该开心吗？」（这不是真正的选择题）
- 识别 XY 问题
  例：用户问「怎么优化 SQL」，实际问题是「schema 设计错了」
- 识别症状 vs 根本原因
  例：「我应该换工作吗？」实际是「我在任何地方都不被尊重」
- 隐性知识提取（Polanyi 协议）
  例：当用户说「就是那种感觉」时，通过例子/反例来澄清
```

#### QuestionOS 当前做法
```
Step 1: 简单的信息密度检查
- 有意义字符数 > 8
- 英文词数 > 5
- 去掉口头禅后仍有内容
→ 仅检查"是否足够具体"，不检查"是否是真问题"
```

**差距**：用户可能进入审议室，但问题本身是伪问题或症状
→ **后果**：审议很难产出有意义的结果

---

### 🔴 差距 2：证据的系统性

#### Agora 做法
```
Step 1: Evidence Gathering（强制）
- 不同房间有不同的证据策略
- Forge（工程）：必须读代码 + git log + 测试
- Bazaar（商业）：市场数据 + 竞品 + 财务
- Oracle（人生）：个人历史 + 价值观声明
- Hearth（关系）：互动历史 + 沟通模式
- Clinic（心理）：行为日志 + 触发因素
- Atelier（创意）：作品样本 + 读者反馈

→ 每个房间都是「证据优先」
```

#### QuestionOS 当前做法
```
→ 没有强制的证据收集步骤
→ Agent 直接基于用户的叙述进行分析
→ 无法验证用户描述的准确性
```

**差距**：分析基于不完整或有偏差的信息
→ **后果**：高层级的意见建立在沙地基础上

---

### 🔴 差距 3：角色的多样性和对立性

#### Agora 做法
```
31 个具体的思想家，每个都有明确的:
- 分析视角
- 会质疑的角度
- 极端立场对

例如 Forge（工程室）的 7 个成员：
- Feynman：「这解释不清楚就不是真的」
- Ada：「形式化系统和可机械化的界限」
- Torvalds：「实用主义 - 做好或闭嘴」
- Popper：「用什么能否被证伪来检验」
- Occam：「每个实体都要为自己辩护」
- Nietzsche：「旧的必须死，新的才能活」
- Wittgenstein：「语言游戏的边界就是世界的边界」

→ 不是「统一方向的建议」，而是「多个极端视角的碰撞」
```

#### QuestionOS 当前做法
```
4 个角色，但都指向同一方向的「帮助用户思考」：
- Auditor（审计者）：质疑假设
- Value Judge（价值判官）：权衡价值
- Risk Officer（风险官）：评估风险
- Integrator（整合者）：综合意见

→ 这些角色有某种「共识」，即「帮助用户做出更好决策」
→ 缺少「根本对立的观点」
```

**差距**：缺乏能够彼此**质疑基本假设**的角色
→ **后果**：
  - 讨论停留在"好的建议"而非"挑战核心信念"
  - 没有人会说"你整个问题框架都错了"
  - 用户无法见证真正的思想碰撞

---

### 🔴 差距 4：立场碰撞的深度

#### Agora 做法
```
Round 1（独立分析）：
- 所有成员独立分析，互不知道他人意见
- 目标：产出最诚实的个人观点，不妥协

Round 2（黑格尔式交叉质询）：
- 重点改变：不再是「推荐一个答案」
- 而是：「提出一个综合」
- 格式：Thesis（多数观点）vs Antithesis（少数观点）vs Synthesis（更高阶整合）

例：
- Thesis：「用微服务，扩展性强」（多数）
- Antithesis：「用单体，运维简单」（少数）
- Synthesis：「在当前阶段用单体 + 清晰的边界，3 年后再分拆」
  （这是一个新想法，不是左右折中，而是上升到了不同维度）
```

#### QuestionOS 当前做法
```
步骤②：轮流发言（4 轮）
- Round 1（Auditor）：质疑
- Round 2（Value Judge）：权衡
- Round 3（Risk Officer）：评估风险
- Round 4（Integrator）：综合意见

→ 无法进行「真正的交叉对话」
→ 最后一轮「综合」缺少前面的对立观点
→ 每个发言是独立的，不是在构建一个思辨的高塔
```

**差距**：缺乏 Round 2 的对抗性和综合性
→ **后果**：
  - 用户不会看到"本来对立的观点如何整合成新想法"
  - 最终意见往往是"折中"而非"突破"

---

### 🔴 差距 5：用户交互的深度和频率

#### Agora 做法
```
AskUserQuestion #1（Step 2）：
- 问题溶解检查（识别伪问题、XY 问题、症状）
- 隐性知识提取（「你说不清楚」→ 例子 → 反例 → 行为）
- 显示各成员的不同理解（有意义的发散）
→ 让用户改进问题定义

AskUserQuestion #2（Step 4）：
- 展示 Round 1 的共识程度
- 如果共识高：「大家都同意，要进一步吗？」
- 如果共识低：「大家意见相反，深入讨论会很有价值」
→ 让用户决定是否需要第 2 轮（自适应深度）

可选：Tacit Knowledge Protocol
- 当用户说「就是个感觉」时，强制进行 3 步提取
```

#### QuestionOS 当前做法
```
Step 1（追问）：
- 通过维度库生成追问，澄清问题
- 1-2 轮后用户进入步骤②
→ 追问是「单向的澄清」，不是「用户决策」

Step 2：
- 无停顿，直接进入 4 轮审议
- 无决策点（用户无法选择深度、成员等）
→ 用户被动接收，不是主动参与
```

**差距**：用户参与度低，无法中途改变方向
→ **后果**：用户可能坐享"完整的审议"，但如果问题有问题也得不到矫正

---

### 🔴 差距 6：模式的灵活性

#### Agora 做法
```
- --full：所有成员（6-7 人）
- --quick：2 轮，无 AskUser，加速版（适合急速决策）
- --duo：2 人对话，用极端对立的观点（适合快速认清两个立场）
- --triad：预定义的 3 人组（针对特定问题类型优化）
- --members name1,name2：自定义（适合已知的冲突对象）
- --depth full/auto：强制第 2 轮或自适应

→ 同一个系统，不同使用场景有不同的"量身定制"
```

#### QuestionOS 当前做法
```
- 沙盘总是「完整的 4 轮」
- 无法快速决策（总要经过完整流程）
- 无法选择参与者（固定 4 个角色）
- 无法选择深度（总是完整）

→ 适合"深思熟虑型"用户，不适合"快速判断型"用户
```

**差距**：一刀切的方案，无法灵活应对不同场景
→ **后果**：
  - 简单问题被强制进行深度分析（浪费时间）
  - 复杂问题无法进行深度分析（解决不了）

---

### 🔴 差距 7：面板选择的智能化

#### Agora 做法
```
Auto-Triad Selection：
1. 读问题
2. 匹配关键词到预定义的 triad
3. 解释选择逻辑
4. 执行

例如：
Q: "Should we refactor this legacy codebase?"
→ 关键词「refactor, legacy」
→ Triad：Nietzsche（创造性破坏） + Occam（精简） + Ada（形式化新系统）
→ 理由陈述给用户，可覆盖
```

#### QuestionOS 当前做法
```
审议室选择基于「分诊结果」：
- BUSINESS → 集市
- ENGINEERING → 锻造坊
- ...

但房间内的「4 个角色」总是固定的。

→ 无法根据问题的具体类型选择不同的「角色组合」
```

**差距**：
- Agora 会问「这是重构问题还是架构问题」，选择不同的 3 人组
- QuestionOS 会问「这是技术问题还是商业问题」，选择不同的 4 人组

→ **后果**：面板不是为问题类型优化的，而是为大类别优化的

## 核心缺陷总结

| 维度 | QuestionOS 当前 | Agora | 影响 |
|------|-----------------|-------|------|
| **问题质量检查** | 信息密度 | 问题溶解 + 隐性知识 | 用户可能进入「假问题」的审议 |
| **证据基础** | 用户叙述 | 系统化证据收集 | 分析基于不完整信息 |
| **角色多样性** | 4 个角色，指向同向 | 31 个思想家，包含极端对立 | 缺乏根本性的思维碰撞 |
| **对话结构** | 轮流发言 | Round 1 + Round 2（黑格尔） | 无法产出真正的「综合」 |
| **用户参与** | 被动接收 | 2 个决策点 + 自适应深度 | 用户无法引导过程 |
| **使用场景** | 一刀切（完整） | 6 种模式（Full/Quick/Duo/Triad...） | 简单问题被过度分析 |
| **面板选择** | 按大类别 | 按具体问题类型 | 面板不够精准 |
| **综合能力** | Integrator 做总结 | Synthesis（上升到新维度） | 结论往往是「折中」而非「突破」 |

## 优化路线图（从简到复杂）

### 第一阶段（低成本，高收益）

**1. 加入问题溶解检查**（可用现有的 MainCalibrateAgent）
```
在步骤①的追问前，加一个检查层：
- 识别伪问题
- 识别 XY 问题
- 识别症状 vs 根本原因
→ 如果有发现，询问用户「真实问题是什么」
```

**2. 加入隐性知识提取**（Polanyi 协议）
```
当用户说「就是那种感觉，说不清楚」时：
- 问：你能举个例子吗？
- 问：反例是什么？
- 问：在什么具体情况下会这样？
→ 把隐性变成显性
```

**3. 加入自适应深度**
```
Step 1 的结尾，评估四个角色的共识度：
- 高共识（>80%）：「大家都同意，要更深入吗？」
- 低共识（<60%）：「大家意见不同，深入会很有价值」
→ 让用户选择是否继续
```

**预期收益**：
- 问题质量 ↑ 30%（减少伪问题的进入）
- 用户体验 ↑（有决策权，感受到被理解）
- 实施难度：中等（改造 SessionService 的流程）

### 第二阶段（中等成本，最高收益）

**4. 加入 Round 2（黑格尔对话）**
```
当前：4 轮轮转发言
→ 改为：Round 1（独立） + Round 2（对话）

Round 2 中：
- 识别 Thesis（多数观点）
- 识别 Antithesis（最强少数观点）
- 要求每个成员提出 Synthesis（新想法）
→ Coordinator 识别最有创意的综合
```

**5. 加入多模式**
```
- --quick：2 轮，加速版
- --deep：强制 Round 2
- --duo：2 人对话
→ 框架层面支持不同的使用场景
```

**6. 优化角色对立性**
```
当前 4 个角色都比较"好人"
→ 加入更极端的观点：
  - 「这个问题根本不应该这么框架」（Nietzsche 式）
  - 「这个决策没有任何依据」（Popper 式）
  - 「你漏掉了什么关键约束」（Occam 式）
```

**预期收益**：
- 见证真正的思想碰撞
- 能产出突破性的综合意见（而非折中）
- 实施难度：高（需要重构 AgentOrchestrator）

### 第三阶段（长期，参考 Agora）

**7. 扩展到 20+ 个具体思想家**
**8. 实现 31 个思想家的完整生态**
**9. 房间级别的面板优化**（按问题类型选择不同的角色组）
**10. 证据驱动的分析框架**
